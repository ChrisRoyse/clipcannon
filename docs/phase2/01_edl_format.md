# EDL Format & Edit Creation

## 1. Overview

The Edit Decision List (EDL) is the bridge between AI understanding and video rendering. The AI reads the Video Understanding Document (VUD) produced by Phase 1, reasons about what makes a compelling clip, and writes a declarative EDL describing the output. The rendering engine then translates that EDL into FFmpeg commands. The AI never touches FFmpeg directly -- it expresses creative intent through structured data.

Key principles:

- **Declarative**: The EDL says *what* the output should look like, not *how* to render it. The rendering engine handles all FFmpeg translation.
- **Source-immutable**: Every EDL references the original source video by SHA-256 hash. The source is never modified. All edits are non-destructive overlays on the original.
- **Self-contained**: A single EDL contains everything needed to produce one output clip -- segments, transitions, captions, crop, audio, overlays, and render settings.
- **Platform-aware**: Each EDL targets a specific platform profile, constraining aspect ratio, duration, and encoding parameters.

Workflow:

```
AI reads VUD (Phase 1)
    |
    v
AI writes EDL (JSON)
    |
    v
clipcannon_create_edit validates + stores EDL
    |
    v
clipcannon_render translates EDL -> FFmpeg -> output.mp4
    |
    v
Human reviews via dashboard -> approve / reject
```

EDLs are stored in the `edits` table in `analysis.db` per project. Each EDL describes one output clip. A single source video can produce unlimited EDLs -- one per clip, one per platform variant, or one per revision.

### 2.1 Top-Level Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `edit_id` | string | Yes | Unique identifier, auto-generated by `clipcannon_create_edit`. Format: `edit_{uuid_short}` |
| `project_id` | string | Yes | References the project this edit belongs to |
| `name` | string | Yes | Human-readable name for the edit (e.g. "TikTok Highlight #3") |
| `created_at` | string (ISO 8601) | Yes | Creation timestamp, auto-set by the server |
| `status` | string enum | Yes | One of: `draft`, `rendering`, `rendered`, `approved`, `rejected` |
| `source_sha256` | string | Yes | SHA-256 hash of the original source video. Verified before every render. |
| `target_platform` | string enum | Yes | One of: `tiktok`, `instagram_reels`, `youtube_shorts`, `youtube_standard`, `youtube_4k`, `facebook`, `linkedin` |
| `target_profile` | string enum | Yes | Encoding profile key. Must match a valid profile from the profiles registry. |
| `segments` | array | Yes | Ordered list of segment objects (at least one required) |
| `captions` | object | Yes | Caption configuration |
| `crop` | object | Yes | Crop and reframing configuration |
| `audio` | object | Yes | Audio mixing configuration |
| `overlays` | object | Yes | Visual overlay configuration |
| `metadata` | object | Yes | Platform metadata (title, description, hashtags) |
| `render_settings` | object | Yes | Encoding and quality settings |

### 2.2 Status Lifecycle

```
draft ──> rendering ──> rendered ──> approved
                   |            └──> rejected
                   └──> failed
```

- **draft**: EDL created and validated, not yet rendered.
- **rendering**: Render job is in progress.
- **rendered**: Render complete, output file available.
- **approved**: Human approved via dashboard. Ready for publishing.
- **rejected**: Human rejected. AI may create a revised EDL.
- **failed**: Render failed (FFmpeg error, source hash mismatch, etc.).


## 3. Segment Model

Each segment represents a contiguous time range extracted from the original source video. Multiple segments are concatenated in output order to form the final clip.

### 3.1 Segment Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `segment_id` | integer | Yes | Sequential ID within this EDL (1-based) |
| `source_start_ms` | integer | Yes | Start time in the original source video (milliseconds) |
| `source_end_ms` | integer | Yes | End time in the original source video (milliseconds) |
| `output_start_ms` | integer | Yes | Where this segment starts in the output timeline (milliseconds) |
| `speed` | float | Yes | Playback speed multiplier. 1.0 = normal, 0.5 = half-speed (slow-mo), 2.0 = double-speed |
| `transition_in` | object or null | Yes | Transition applied at the start of this segment |
| `transition_out` | object or null | Yes | Transition applied at the end of this segment |

### 3.2 Time Calculations

The effective duration of a segment in the output is:

```
source_duration_ms = source_end_ms - source_start_ms
output_duration_ms = source_duration_ms / speed
```

For example, a 30-second source segment at 0.5x speed produces 60 seconds of output. At 2.0x speed, it produces 15 seconds.

Speed constraints:
- Minimum: 0.25 (quarter speed)
- Maximum: 4.0 (4x speed)
- 1.0 is the default and most common value

### 3.3 Transition Objects

Transitions are applied between segments. A `transition_out` on segment N overlaps with the `transition_in` on segment N+1.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string enum | Yes | Transition type |
| `duration_ms` | integer | Yes | Duration of the transition effect |

Supported transition types:

| Type | Description | Typical Duration |
|------|-------------|-----------------|
| `fade` | Fade to/from black | 300-1000ms |
| `crossfade` | Dissolve between segments | 500-1500ms |
| `wipe_left` | Wipe from right to left | 300-800ms |
| `wipe_right` | Wipe from left to right | 300-800ms |
| `wipe_up` | Wipe from bottom to top | 300-800ms |
| `wipe_down` | Wipe from top to bottom | 300-800ms |
| `slide_left` | Slide new segment in from right | 300-800ms |
| `slide_right` | Slide new segment in from left | 300-800ms |
| `dissolve` | Gradual blend (alias for crossfade) | 500-1500ms |
| `zoom_in` | Zoom into next segment | 300-600ms |
| `cut` | Hard cut with no effect (explicit) | 0ms |

Transition duration constraints:
- Minimum: 100ms
- Maximum: 2000ms
- Must not exceed 50% of the shorter adjacent segment's output duration

### 3.4 Segment Ordering

Segments in the `segments` array define the output timeline. Rules:

1. Segments are ordered by `output_start_ms` (ascending).
2. Segments must not overlap in the output timeline (except for transition overlap regions).
3. Gaps between segments are filled with black frames (or transition effects).
4. `output_start_ms` of the first segment is typically 0 but can be non-zero (e.g., to leave room for a title card).


## 4. Captions Object

Captions are burned into the video during rendering. Caption text and timing are derived from the `transcript_words` table (Phase 1 WhisperX data).

### 4.1 Caption Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `enabled` | boolean | Yes | Whether to render captions |
| `style` | string enum | Yes | Visual style preset |
| `font` | string | Yes | Font family name (must be installed on system) |
| `font_size` | integer | Yes | Font size in pixels |
| `color` | string | Yes | Text color (hex, e.g. `#FFFFFF`) |
| `stroke_color` | string | Yes | Text outline/stroke color (hex) |
| `stroke_width` | integer | Yes | Stroke width in pixels |
| `position` | string enum | Yes | Vertical position: `center`, `bottom`, `top` |
| `animation` | string enum | Yes | Animation style: `word_by_word`, `fade`, `none` |
| `chunks` | array | No | Pre-computed caption chunks. Auto-generated from `transcript_words` if empty. |

### 4.2 Caption Styles

| Style | Description | Best For |
|-------|-------------|----------|
| `bold_centered` | Large bold text centered on screen, word-by-word highlight | TikTok, YouTube Shorts |
| `word_highlight` | Current word highlighted in accent color, others dimmed | Instagram Reels |
| `subtitle_bar` | Semi-transparent bar at bottom with subtitle text | YouTube Standard, LinkedIn |
| `karaoke` | Words light up progressively as spoken | Music-heavy content |

### 4.3 Caption Chunks

When `captions.enabled` is true and `chunks` is empty, `clipcannon_create_edit` auto-generates chunks from the `transcript_words` table for the time ranges covered by the EDL's segments. Each chunk is a group of 3-7 words displayed simultaneously:

```json
{
  "chunk_id": 1,
  "text": "The one thing most",
  "start_ms": 0,
  "end_ms": 1200,
  "words": [
    {"word": "The", "start_ms": 0, "end_ms": 200},
    {"word": "one", "start_ms": 210, "end_ms": 380},
    {"word": "thing", "start_ms": 400, "end_ms": 580},
    {"word": "most", "start_ms": 600, "end_ms": 800}
  ]
}
```

Chunk timings are relative to the output timeline (not the source). The rendering engine maps source timestamps to output timestamps using the segment definitions.


## 5. Crop Object

Crop controls how the source frame (typically 16:9) is reframed to the target aspect ratio (e.g., 9:16 for vertical platforms).

### 5.1 Crop Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `mode` | string enum | Yes | `auto`, `manual`, `none` |
| `aspect_ratio` | string | Yes | Target aspect ratio: `9:16`, `1:1`, `16:9`, `4:5` |
| `face_tracking` | boolean | Yes | Enable MediaPipe face detection to center crop on faces |
| `safe_area_pct` | float | Yes | Percentage of frame width/height that must remain visible (0.0-1.0) |

### 5.2 Crop Modes

- **auto**: The renderer uses face detection (if `face_tracking` is true) and the `shot_type` data from Phase 1 to determine optimal crop position per frame. Crop position smoothly interpolates between keyframes to avoid jitter.
- **manual**: The AI specifies explicit crop coordinates (future extension -- not implemented in initial Phase 2).
- **none**: No cropping applied. Output uses source frame as-is (only valid when source aspect ratio matches target).

### 5.3 Face Tracking Behavior

When `face_tracking` is true:

1. MediaPipe detects face bounding boxes in sampled frames (reusing Phase 1's 2fps extracted frames where available).
2. The crop window is positioned to keep the primary face within the `safe_area_pct` region.
3. When multiple faces are detected, the crop favors the currently speaking face (using speaker diarization data from Phase 1).
4. Crop position is smoothed with exponential moving average to prevent jitter (tau = 500ms).
5. Falls back to center crop when no face is detected.


## 6. Audio Object

Audio controls source audio processing and overlay mixing.

### 6.1 Audio Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `source_audio` | boolean | Yes | Include audio from the source video |
| `source_volume_db` | float | Yes | Volume adjustment for source audio (0 = unchanged, -6 = quieter, +3 = louder) |
| `background_music` | object or null | Yes | Background music specification (see Audio Generation spec) |
| `sound_effects` | array | Yes | List of sound effect objects to overlay |
| `ducking` | object | Yes | Speech-aware audio ducking configuration |

### 6.2 Background Music Object

When `background_music` is non-null, it specifies either a reference to a previously generated music asset or parameters for on-demand generation:

```json
{
  "source": "generated",
  "asset_id": "music_001",
  "volume_db": -18,
  "fade_in_ms": 2000,
  "fade_out_ms": 3000,
  "loop": true
}
```

Or for on-demand generation:

```json
{
  "source": "generate",
  "prompt": "upbeat lo-fi hip hop, 90 BPM, positive energy",
  "duration_ms": 60000,
  "volume_db": -18,
  "fade_in_ms": 2000,
  "fade_out_ms": 3000,
  "loop": false
}
```

### 6.3 Sound Effect Object

```json
{
  "type": "transition_whoosh",
  "source": "programmatic_dsp",
  "trigger_ms": 45000,
  "duration_ms": 500,
  "volume_db": -12,
  "params": {
    "sweep_hz_start": 200,
    "sweep_hz_end": 8000,
    "decay_rate": 3.0,
    "method": "logarithmic"
  }
}
```

Supported DSP sound effect types: `whoosh`, `riser`, `downer`, `impact`, `chime`, `tick`, `bass_drop`, `shimmer`, `dramatic_stinger`. See the Audio Generation spec (05_audio_generation.md) for full parameter details.

### 6.4 Ducking Object

Audio ducking automatically lowers background music volume when speech is detected. Uses Silero VAD timestamps from the Phase 1 understanding pipeline.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `enabled` | boolean | Yes | Enable speech-aware ducking |
| `duck_level_db` | float | Yes | Volume reduction during speech (negative dB, e.g. -6) |
| `attack_ms` | integer | Yes | Ramp-down time before speech starts (typically 200ms) |
| `release_ms` | integer | Yes | Ramp-up time after speech ends (typically 300ms) |

## 8. Render Settings Object

| Field | Type | Required | Description |

|---------|-----------|--------|-------|---------|-----|-------|----------|

| `instagram_reel` | 1080x1920 | 9:16 | h264_nvenc p4 | 6M | 30 | AAC 192k 44.1kHz | Instagram Reels |
| `youtube_short` | 1080x1920 | 9:16 | h264_nvenc p4 | 8M | 30 | AAC 192k 48kHz | YouTube Shorts |
| `youtube_standard` | 1920x1080 | 16:9 | h264_nvenc p5 | 12M | 30 | AAC 256k 48kHz | YouTube long-form |
| `youtube_4k` | 3840x2160 | 16:9 | h264_nvenc p5 | 40M | 30 | AAC 256k 48kHz | YouTube 4K |
| `linkedin_square` | 1080x1080 | 1:1 | h264_nvenc p4 | 5M | 30 | AAC 192k 44.1kHz | LinkedIn |

### 8.2 Quality Tiers

| Quality | Bitrate Multiplier | NVENC Preset | Use Case |
|---------|-------------------|--------------|----------|
| `high` | 1.0x (profile default) | p5 (slower, better) | Final render for publishing |
| `medium` | 0.7x | p4 (balanced) | Preview for review |
| `draft` | 0.4x | p1 (fastest) | Quick preview during editing iteration |


## 9. Validation Rules

`clipcannon_create_edit` runs a multi-layer validation on every EDL before storing it. Validation failures return detailed error messages so the AI can correct and retry.

### 9.1 Source Validation

- `source_sha256` must match the `source_sha256` stored in the project's `project` table.
- If the project has `vfr_normalized = true`, the renderer uses the CFR-normalized copy (`source_cfr_path`) but the EDL still references the original source hash.

### 9.2 Segment Validation

- `source_start_ms` must be >= 0.
- `source_end_ms` must be <= `duration_ms` from the `project` table.
- `source_end_ms` must be > `source_start_ms`.
- Minimum segment duration: 500ms (before speed adjustment).
- Segments must not overlap in the output timeline (accounting for transition overlap).
- `output_start_ms` values must be in ascending order.
- Speed must be in range [0.25, 4.0].

### 9.3 Timestamp Snapping

Before storing the EDL, all segment timestamps are run through the 3-layer timestamp validator (PRD Section 8.5):

1. **Bounds check**: Reject out-of-bounds timestamps.
2. **Silence snap**: Snap to nearest silence gap within +/-300ms (best audio cut quality).
3. **Word boundary snap**: Snap start to end of preceding word, end to start of following word (no mid-word cuts).
4. **Frame snap**: Snap to nearest extracted frame timestamp (500ms intervals at 2fps).

The snapped timestamps replace the original values. The validation response includes the original and snapped values with drift amounts, so the AI can verify the adjustments are acceptable.

### 9.4 Platform Validation

- Total output duration must be within platform limits:

| Platform | Min Duration | Max Duration | Optimal Duration |
|----------|-------------|-------------|-----------------|
| TikTok | 5s | 180s | 15-60s |
| Instagram Reels | 5s | 90s | 30-45s |
| YouTube Shorts | 5s | 180s | 15-60s |
| YouTube Standard | 30s | 600s (config) | 480-720s |
| Facebook Reels | 5s | 90s | 15-60s |
| LinkedIn | 10s | 600s (config) | 60-90s |

- Aspect ratio in `crop` must be compatible with `target_profile`.
- Font specified in `captions.font` must be available on the system.

### 9.5 Referential Integrity

- `project_id` must reference an existing project with status `analyzed` (Phase 1 pipeline complete).
- If `captions.enabled` is true, the project must have completed the `transcription` stream (verified via `stream_status` table).
- If `crop.face_tracking` is true, the project must have completed the `visual` stream.


## 10. EDL Storage

### 10.1 Database Table

EDLs are stored in the `edits` table in each project's `analysis.db`:

```sql
CREATE TABLE edits (
    edit_id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    edl TEXT NOT NULL,               -- Full EDL as JSON
    platform TEXT,
    profile TEXT,
    status TEXT DEFAULT 'draft',     -- draft, rendering, rendered, approved, rejected, failed
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (project_id) REFERENCES project(project_id)
);
```

The `edl` column stores the complete EDL JSON blob. The `platform` and `profile` columns are denormalized from the EDL for efficient filtering.

### 10.2 Render Linkage

Each edit links to zero or more renders via the `renders` table:

```sql
CREATE TABLE renders (
    render_id TEXT PRIMARY KEY,
    edit_id TEXT NOT NULL,
    output_path TEXT,
    output_sha256 TEXT,
    thumbnail_path TEXT,
    duration_ms INTEGER,
    file_size_bytes INTEGER,
    status TEXT DEFAULT 'pending',
    started_at TEXT,
    completed_at TEXT,
    FOREIGN KEY (edit_id) REFERENCES edits(edit_id)
);
```

When an EDL is rendered, a row is inserted into `renders` with status `pending`, updated to `completed` or `failed` when the render finishes.


## 11. Generation Loss Prevention

This is a critical architectural constraint (PRD Section 8.8). ClipCannon prevents generation loss through five enforcement mechanisms:

### 11.1 Immutable Source Reference

Every EDL contains `source_sha256` -- the SHA-256 hash computed during the Phase 1 probe step. This hash is verified before every render:

```python
# Pseudo-code: renderer verifies source before rendering
source_hash = compute_sha256(source_path)
if source_hash != edl["source_sha256"]:
    raise SourceIntegrityError(
        "Source file has been modified or replaced. "
        "Expected: {edl['source_sha256']}, Got: {source_hash}"
    )
```

### 11.2 Rendered Output Isolation

All rendered outputs are written to `renders/{render_id}/output.mp4`. The rendering engine refuses to open any file from a `renders/` directory as a source input. This makes it physically impossible to re-encode a rendered output.

### 11.3 Single-Pass Rendering

When an EDL specifies multiple segments, the typed-ffmpeg translation layer builds ONE `filter_complex` command that reads all segments from the original source and concatenates in a single encode pass. There are no intermediate rendered segments that get re-encoded.

```
Source ──[segment 1]──┐
         [segment 2]──┤──> filter_complex ──> SINGLE NVENC encode ──> output.mp4
         [segment 3]──┘
```

### 11.4 Re-render from Source

When the AI modifies an EDL and re-renders, it creates a new render job that reads from the original source. The previous render output is not touched -- it can be deleted (ephemeral tier per disk management) or kept for comparison.

### 11.5 Provenance Enforcement

Every render provenance record contains `input.sha256` pointing to the original source's hash. If a render record's input hash matches any output hash from a prior render, the provenance chain integrity check fails -- detecting accidental generation loss.


## 12. MCP Tools

### 12.1 clipcannon_create_edit

Creates a new EDL, validates it, and stores it in the database.

**Parameters:**

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `project_id` | string | Yes | Project to create the edit for |
| `name` | string | Yes | Human-readable name for the edit |
| `target_platform` | string | Yes | Target platform (tiktok, instagram_reels, youtube_shorts, youtube_standard, youtube_4k, facebook, linkedin) |
| `segments` | array | Yes | Segment definitions (source_start_ms, source_end_ms, speed, transitions) |
| `captions` | object | No | Caption configuration (default: enabled with platform-appropriate style) |
| `crop` | object | No | Crop configuration (default: auto with platform-appropriate aspect ratio) |
| `audio` | object | No | Audio configuration (default: source audio only, ducking enabled) |
| `overlays` | object | No | Overlay configuration (default: none) |
| `metadata` | object | No | Title, description, hashtags (default: empty, can be generated later via clipcannon_generate_metadata) |
| `render_settings` | object | No | Render quality and profile (default: profile from target_platform, quality=high) |

**Behavior:**

1. Validates all parameters against the rules in Section 9.
2. Runs timestamp snapping on all segment boundaries.
3. Auto-generates caption chunks from `transcript_words` if `captions.enabled` is true and `chunks` is empty.
4. Auto-fills `source_sha256` from the project's `project` table.
5. Auto-fills `target_profile` from `target_platform` if not explicitly provided.
6. Generates `edit_id` as `edit_{uuid_short}`.
7. Stores the complete EDL in the `edits` table.

**Returns:**

```json
{
  "edit_id": "edit_a1b2c3",
  "status": "draft",
  "segments_count": 2,
  "total_output_duration_ms": 75000,
```

### 12.2 clipcannon_modify_edit

Applies partial updates to an existing EDL.

**Parameters:**

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `project_id` | string | Yes | Project ID |
| `edit_id` | string | Yes | Edit to modify |
| `changes` | object | Yes | Partial EDL object -- only provided fields are changed |

**Behavior:**

1. Loads the existing EDL from the database.
2. Deep-merges `changes` into the existing EDL. Array fields (segments, sound_effects, animations) are replaced entirely, not appended.
3. Re-validates the modified EDL against all rules in Section 9.
4. Re-runs timestamp snapping if any segment timestamps changed.
5. Re-generates caption chunks if segment time ranges changed and `captions.enabled` is true.
6. Resets status to `draft` (any previous render is obsoleted).

**Returns:** Same format as `clipcannon_create_edit`.

## 13. AI Workflow

This is the end-to-end workflow for how the AI uses EDLs to produce edited clips.

### Step 1: Understand the Video

```
AI calls clipcannon_get_vud_summary(project_id)
    --> Gets: duration, speakers, topic list, top highlights, content rating
    --> AI now knows the video's structure and best moments
```

### Step 2: Analyze for Editing

```
AI calls clipcannon_get_analytics(project_id)
    --> Gets: all scenes, topics with boundaries, highlights with scores,
             reactions, silence gaps, beats, quality assessments, pacing data
    --> AI now has the structural map for editing decisions
```

### Step 3: Deep-Dive on Target Sections

```
AI calls clipcannon_get_transcript(project_id, start_ms=845000, end_ms=900000)
    --> Gets: word-level transcript for the time range it plans to clip
AI calls clipcannon_get_segment_detail(project_id, start_ms=845000, end_ms=900000)
    --> Gets: emotion curve, speakers, reactions, beats, pacing for fine-grained decisions
AI calls clipcannon_get_storyboard(project_id, start_ms=845000, end_ms=900000)
    --> AI visually inspects the frames to verify content matches its understanding
```

### Step 4: Create the EDL

```
AI calls clipcannon_create_edit(
    project_id=...,
    name="Best Moment Highlight",
    target_platform="tiktok",
    segments=[
        {source_start_ms: 845000, source_end_ms: 890000, speed: 1.0, ...},
        {source_start_ms: 1230000, source_end_ms: 1260000, speed: 1.0, ...}
    ],
    captions={enabled: true, style: "bold_centered", ...},
    crop={mode: "auto", aspect_ratio: "9:16", face_tracking: true, ...},
    audio={source_audio: true, ducking: {enabled: true, ...}, ...}
)
    --> Server validates, snaps timestamps, generates caption chunks
    --> Returns edit_id
```

### Step 5: Render

```
AI calls clipcannon_render(edit_id=...)
    --> Server translates EDL to FFmpeg filter_complex
    --> Single-pass NVENC encode from original source
    --> Output written to renders/{render_id}/output.mp4
```

### Step 6: Human Review

```
Human reviews rendered clip via dashboard
    --> Approve: status -> "approved", ready for publishing
    --> Reject: AI may modify the EDL and re-render
```

The AI can repeat Steps 3-5 for every clip it wants to produce. After the initial understanding phase (Steps 1-2), the VUD persists in the AI's context. Each subsequent edit draws from the same understanding -- the AI does not re-analyze the video.
