## 1. Executive Summary

**Product Name:** North Star Refinery
**Target User:** Lead Data Scientist / System Architect
**Purpose:** A dedicated standalone GUI application designed to ingest raw raw psychological data, compute "North Star" centroids for the 7 Primary Hogshead Archetypes, mathematically derive the 42 Secondary Archetypes, and fine-tune these coordinates in 5-dimensional embedding space.
**Output:** The authoritative `hogshead_reference_vectors.bin` file used by the Cognitive Radar runtime.

---

## 1.1 Problem Statement

### Current State
The Cognitive Radar runtime requires pre-computed Reference Vectors (North Stars) for 49 Hogshead Archetypes and 3 VAK modalities. Currently, creating these vectors requires:
- Manual collection and organization of training audio/text samples
- Custom Python scripts for embedding extraction
- Ad-hoc statistical calculations for centroid computation
- No visualization of vector quality or cluster separation
- No validation mechanism before runtime deployment

### Pain Points

| Pain Point | Impact | Frequency |
|:-----------|:-------|:----------|
| Manual audio editing in external tools (Audacity) | 2-4 hours per archetype | Every calibration |
| No way to detect "toxic" samples skewing centroids | Corrupted Reference Vectors → runtime misclassification | Unknown until runtime failure |
| No visualization of 768-dim vectors | Cannot verify cluster quality | Every calibration |
| Binary export requires manual struct alignment | Runtime crashes from schema mismatch | ~30% of exports |
| No reproducibility guarantee | Different runs produce different vectors | Every run |
| No VAK profile synthesis capability | Missing modality detection in runtime | Not currently possible |

### Opportunity
A dedicated calibration tool that:
1. **Reduces calibration time** from 8-12 hours to <1 hour through automation
2. **Eliminates human error** via statistical outlier detection and validation gates
3. **Provides mathematical proof** of vector quality before deployment
4. **Ensures bitwise reproducibility** for regulatory compliance
5. **Synthesizes VAK profiles** from archetype training data
6. **Exemplar Calibration**: Ingest successful debt resolution call recordings to create "High-EQ" North Stars. This ensures reference vectors represent the ideal version of an archetype (e.g., "Competent Power" rather than "Aggressive Power") to guide the runtime toward resolution rather than simple mirroring.

### Consequences of Not Solving
- Runtime misclassification rates >15% (currently acceptable: <5%)
- No way to audit or reproduce calibration decisions
- Manual process blocks scaling to new domains/languages
- No VAK modality support in production

**"Glass Box" Data Science:** Unlike the runtime system which automates decisions, this tool must expose **every intermediate float value**. The user must see the raw embeddings, the averaged centroids, the distance matrices, and the mixing weights.

## 3. Functional Requirements

### 3.1 Module A: The Ingestion Engine
**Goal:** Easy loading of raw ground-truth data for the 7 Primary Advantages.

*   **REQ-ING-001: Structured Import with Gender Tagging.**
    *   UI must provide 7 distinct "Buckets" (Power, Passion, Prestige, Trust, Mystique, Innovation, Alert).
    *   Each Bucket accepts two streams: **Audio Files** (.wav, .mp3) and **Text Data** (.txt, .json).
    *   **Gender Tagging for Audio Files:**
        *   The UI for Audio Files ingestion MUST allow the user to tag files as **Male** or **Female**.
        *   **Auto-Detection Option:** The system provides a "Detect Gender" button that runs the WavLM-based gender classifier on the audio file and auto-populates the gender tag.
        *   **Manual Override:** The user can manually override the auto-detected gender tag if the classifier is incorrect.
        *   **Validation:** Audio files without a gender tag MUST NOT be processed. The UI displays a warning: *"Gender tag required for audio files. Select Male or Female."*
        *   **Text Files:** Text files remain gender-neutral and do not require gender tagging.
*   **REQ-ING-002: Mirror Preprocessing.**
    *   **CRITICAL:** The Refinery MUST use the *exact* same Audio Preprocessor (VAD, Normalization, Resampling) and Text Preprocessor as the runtime Cognitive Radar.
    *   *Validation:* UI must show "Preprocessed Waveform" vs "Raw Waveform" to confirm VAD is working.
*   **REQ-ING-002a: cuFFT DX Audio Pipeline.**
    *   **Feature Used:** *cuFFT Device Extensions (Blackwell Performance Update)*.
    *   **Implementation:** Perform Spectrogram extraction and VAD energy calculations using the updated Blackwell-optimized cuFFT libraries directly in device memory.
    *   **Benefit:** Zero-copy audio processing. Audio loads from disk directly to GPU memory (via GDS if available, or pinned memory) and is processed without CPU round-trips.
*   **REQ-ING-003: Batch Vectorization.**
    *   System runs inference on all imported files using the 5 core models (Nomic, WavLM, Wav2Vec2, RoBERTa, BERT).
    *   UI displays a progress bar for GPU batch processing.
*   **REQ-ING-004: Synthetic Data Generator (The "Lazy Button").**
    *   **Problem Solved:** Writing 100 sentences for "Prestige" vs "Power" is tedious.
    *   **Solution:** A built-in prompt interface to Qwen (local LLM) to generate training data.
    *   **Input:** User selects an Archetype (e.g., "Mystique") and Context (e.g., "Debt Negotiation").
    *   **Action:** System prompts Qwen: *"Generate 50 distinct, short sentences that a 'Mystique' personality would say during a successful debt negotiation where they de-escalate tension while maintaining firm boundaries."*
    *   **Review:** User reviews the list, unchecks "bad" generations, and clicks "Import to Stream A."
    *   **Benefit:** Generates 100s of samples in seconds instead of manual typing. Training data focuses on resolution-oriented, high-EQ exemplars rather than generic personality traits.
*   **REQ-ING-005: Smart Audio Slicer (Auto-Segmenter).**
    *   **Problem Solved:** Manual audio editing in Audacity to chop 20 minutes of interviews into 5-second clips.
    *   **Solution:** Upload one long file, and the system chops it automatically.
    *   **Action:** When a long audio file (>10s) is dropped, the system runs VAD (Voice Activity Detection).
    *   **Result:** Slices audio into "Speech Chunks" (3-7 seconds long).
    *   **UI:** Shows a "Filmstrip" of chunks. User can delete coughing/silence, then click "Import All."
    *   **Benefit:** Ingests raw podcasts/interviews instantly.
*   **REQ-ING-006: Data Augmentation Pipeline (Robustness Training).**
    *   **Problem Solved:** Training data is typically clean studio audio. Real users will be on noisy microphones with background interference. Clean-trained North Stars fail to match noisy real-world input.
    *   **Solution:** A checkbox "Augment Data" during ingestion.
    *   **Logic:** For every clean audio file, generate 3 variants:
        1.  **Low Noise:** Add 5% white noise.
        2.  **Cafe Noise:** Overlay faint background chatter.
        3.  **Gain Shift:** Reduce volume by -6dB.
    *   **Math:** The North Star Centroid is calculated using the Clean + Augmented versions together.
    *   **Benefit:** Creates "Heavy" Anchors that are stable even when the user is in a noisy environment.

*   **REQ-ING-007: Chronemic Feature Extraction Pipeline.**
    *   **Context:** Unlike embeddings which come from neural models, Chronemics are computed from metadata (timestamps and transcripts). This is the **6th Embedding Stream** required by the Runtime.
    *   **Hardware:** Executes on **Green Context A** (Audio Lane) using the VAD output. Zero-copy memory access allows timestamp calculation during the VAD pass.
    *   **Calculation Logic (Per Audio Sample):**
        1.  **Input:** Audio Waveform + Text Transcript + VAD Timestamps (Start/End of speech chunks).
        2.  **WPM (Words Per Minute):**
            *   Formula: `(Transcript Word Count / Total Speech Duration in Minutes)`
            *   **CRITICAL:** Use *speech duration* (sum of VAD chunks), NOT total audio duration. Silence must not skew WPM.
            *   Example: 150 words in 60 seconds of speech = 150 WPM (even if audio file is 90 seconds with 30 seconds of silence).
        3.  **Latency (Average Silence Gap):**
            *   Calculate the duration of all gaps between VAD Speech Chunks.
            *   Filter: Only count gaps > 200ms (ignore micro-pauses from breathing).
            *   Return the `Mean` of qualifying gaps in milliseconds.
            *   Example: Gaps of [50ms, 320ms, 180ms, 450ms] → Mean of [320ms, 450ms] = 385ms.
        4.  **Jitter (Pace Variance):**
            *   Calculate the duration of every Speech Chunk from VAD.
            *   Return the `Standard Deviation` of these durations.
            *   High jitter = erratic/anxious speech (Power/Alert archetypes).
            *   Low jitter = steady/measured speech (Trust/Mystique archetypes).
        5.  **Overlap Ratio:**
            *   For Reference Training Data: Force to `0.0` (single-speaker monologues).
            *   For Runtime (multi-speaker): Calculate as `(overlapping_speech_ms / total_speech_ms)`.
    *   **Output:** Appends `[f32; 4]` vector (`[wpm, latency, jitter, overlap]`) to the `RawSample` struct.
    *   **UI Display:** Show a "Chronemic Stats" panel in the Source Inspector:
        *   **Speed Meter:** Visual gauge showing WPM (color-coded: <100=Slow/Blue, 100-160=Normal/Green, >160=Fast/Red).
        *   **Silence Pattern:** Bar chart showing distribution of pause durations.
    *   **Validation:**
        *   If WPM < 50 or WPM > 300, flag sample as "Anomalous Timing" (likely VAD error or non-speech audio).
        *   If Latency > 5000ms (5 seconds average), flag as "High Latency Warning".

*   **REQ-ING-007a: Chronemic Extraction CUDA Implementation.**
    *   **Feature Used:** *Green Context A (Audio Lane)* with zero-copy memory access.
    *   **Pipeline Integration:** Chronemic calculations occur *inline* with VAD processing:
        1.  VAD kernel outputs speech chunk timestamps to shared memory.
        2.  Chronemic kernel (same SM partition) immediately consumes timestamps.
        3.  No PCIe round-trip required.
    *   **Kernel Pseudocode:**
    ```cuda
    __global__ void chronemic_extract(
        const VADTimestamp* chunks,
        int chunk_count,
        const char* transcript,
        int word_count,
        ChronemicVector* output
    ) {
        // Calculate total speech duration (parallel reduction)
        float total_speech_ms = parallel_sum(chunks, chunk_count);

        // WPM = (word_count / total_speech_ms) * 60000
        output->wpm = (word_count / total_speech_ms) * 60000.0f;

        // Calculate gaps between chunks
        float* gaps = extract_gaps(chunks, chunk_count);
        float* filtered_gaps = filter_gaps_gt(gaps, 200.0f); // >200ms only
        output->latency = parallel_mean(filtered_gaps);

        // Jitter = std_dev of chunk durations
        float* durations = extract_durations(chunks, chunk_count);
        output->jitter = parallel_std_dev(durations);

        // Overlap forced to 0.0 for training data
        output->overlap = 0.0f;
    }
    ```
    *   **Benefit:** Chronemic extraction adds <1ms to the audio processing pipeline.

### 3.2 Module B: The Forge (Calculation Core)
**Goal:** Calculate centroids, mathematically generate secondary profiles, compute cultural baselines, and synthesize VAK reference vectors.

*   **REQ-FORGE-001: Primary Centroid Calculation (6-Stream with Gender Separation).**
    *   For each of the 7 Buckets, calculate the Mean Vector for all **6** embedding spaces (Semantic, Tone, Emotion, Style, Action, **Chronemic**).
    *   **Gender-Specific Centroid Calculation for Audio Streams:**
        *   For **Tone (Stream B)** and **Emotion (Stream C)**, the system MUST calculate **two separate centroids** per Bucket:
            *   **Male Centroid:** Derived exclusively from samples tagged as `Gender::Male`.
            *   **Female Centroid:** Derived exclusively from samples tagged as `Gender::Female`.
        *   **Semantic, Style, Action, and Chronemic streams** calculate a single unified centroid (gender-neutral).
    *   **Minimum Sample Requirement per Gender:**
        *   Each gender category within a Bucket MUST have at least 3 samples for centroid calculation.
        *   If a gender category has fewer than 3 samples, the UI displays: *"Warning: Insufficient [Male/Female] samples in [Bucket]. Add at least 3 samples for reliable centroid."*
    *   **Chronemic Centroid Calculation:**
        *   Calculate mean and standard deviation for each of the 4 chronemic dimensions (WPM, Latency, Jitter, Overlap) across all samples in the bucket.
        *   Store the mean values as the centroid's chronemic vector.
    *   *Visual:* Show standard deviation/variance within the bucket for each gender. If "Power Male" audio samples have high variance, the centroid is unstable (Data Quality Warning).

**REQ-FORGE-001a: Gender-Specific Centroid Storage Structure**

The calculated centroids are stored in the `NorthStarReferenceSet` struct with gender-specific fields for audio streams:

```rust
/// Centroid calculation result for a single Bucket (Primary Advantage)
pub struct BucketCentroidResult {
    pub advantage: Advantage,

    // Gender-Neutral Streams (calculated from all samples)
    pub semantic_centroid: [f32; 768],
    pub style_centroid: f32,
    pub action_centroid: [f32; 42],
    pub chronemic_centroid: [f32; 4],

    // Gender-Specific Audio Streams
    pub tone_centroid_male: [f32; 512],
    pub tone_centroid_female: [f32; 512],
    pub emotion_centroid_male: [f32; 1024],
    pub emotion_centroid_female: [f32; 1024],

    // Quality metrics per gender
    pub male_sample_count: u32,
    pub female_sample_count: u32,
    pub tone_variance_male: f32,
    pub tone_variance_female: f32,
    pub emotion_variance_male: f32,
    pub emotion_variance_female: f32,
}
```
*   **REQ-FORGE-002: The Mixing Matrix (49 Generator) (Tile Accelerated).**
    *   Generate the 42 Secondary Archetypes using Weighted Vector Addition.
    *   **Formula:** `Target = Normalize( (Primary * W1) + (Secondary * W2) )`.
    *   *UI Control:* "Mixing Sliders." Allow the Data Scientist to adjust `W1` (default 0.7) and `W2` (default 0.3) globally or per-archetype to fine-tune the blend.
*   **REQ-FORGE-002a: Tile-Based Vector Mixing.**
    *   **Feature Used:** *CUDA Tile IR*.
    *   **Logic:** Instead of looping through vectors with standard SIMT threads, map the 7 Primary Reference Vectors as "Tiles" in the L2 Cache.
    *   **Operation:** Perform the weighted mix (`0.7*P + 0.3*S`) using Tile-level instructions.
    *   **Benefit:** Extremely fast "Recalculate" speeds. When you move a slider in the UI, the 49 North Stars update in real-time (sub-millisecond latency) because the data never leaves the GPU registers/cache.
*   **REQ-FORGE-003: Z-Score Outlier Auto-Pruning (Toxic Sample Detector).**
    *   **Problem Solved:** One bad file (background noise, actor stuttering, wrong tone) in a batch of 50 skews the centroid significantly. Finding that one file manually is impossible.
    *   **Solution:** Automated statistical outlier detection before centroid finalization.
    *   **Logic:** Before finalizing a centroid, the system calculates a temporary centroid. It then measures the distance of every sample from that center.
    *   **Action:** Any sample with a distance > **2.5 Standard Deviations** (Sigma) from the mean is automatically flagged as "Toxic."
    *   **UI:** These files are highlighted in **Red** in the Source list with an "Exclude?" toggle.
    *   **Benefit:** Keeps North Stars pure without manual hunting for bad samples.
*   **REQ-FORGE-004: Bitwise Reproducibility.**
    *   **Feature Used:** *CCCL 3.1 Deterministic Reductions*.
    *   **Problem:** Floating point math is non-associative. Parallel reduction (calculating the mean of 100 vectors) can result in slightly different North Star coordinates run-to-run. This is unacceptable for a "Standard."
    *   **Implementation:** Use `cub::DeviceReduce` with the `Run-to-run` determinism guarantee flag.
    *   **Benefit:** Your North Star coordinates are mathematically absolute constants.

*   **REQ-FORGE-005: Cultural Baseline Extraction (Z-Score Prep).**
    *   **Problem:** The Runtime needs to know what "Normal" is to calculate Z-Scores using the formula: `Z = (Value - Mean) / StdDev`. Without baseline statistics, the Runtime cannot normalize chronemic data for cross-cultural comparison.
    *   **Action:** After all samples are ingested, the Forge MUST calculate the **Global Mean** and **Global Standard Deviation** for each chronemic dimension across the *entire dataset*.
    *   **Calculation Logic:**
        1.  Collect all WPM values from every ingested sample (regardless of archetype).
        2.  Calculate: `mean_wpm = sum(all_wpm) / count`
        3.  Calculate: `std_dev_wpm = sqrt(sum((wpm - mean_wpm)^2) / count)`
        4.  Repeat for Latency, Jitter (Overlap is always 0.0 for training).
    *   **Output:** A `CulturalBaseline` struct containing:
        ```rust
        CulturalBaseline {
            mean_wpm: f32,        // e.g., 145.0
            std_dev_wpm: f32,     // e.g., 28.5
            mean_latency: f32,    // e.g., 450.0 ms
            std_dev_latency: f32, // e.g., 180.0 ms
            mean_jitter: f32,     // e.g., 0.35
            std_dev_jitter: f32,  // e.g., 0.12
            sample_count: u32,    // Total samples used
        }
        ```
    *   **UI Panel:** "Cultural Baseline Stats" in The Forge tab:
        *   Display calculated Global Mean WPM (e.g., "145 WPM") with visual range indicator.
        *   Display Global Mean Latency (e.g., "450ms average silence").
        *   **Override Controls:** Allow the Data Scientist to manually override these values if the training data is biased:
            *   "Force Baseline WPM to: [___]" (e.g., set to 150 if you know your training data speaks faster than the general population).
            *   "Force Baseline Latency to: [___]" (e.g., set to 400ms for faster-paced cultures).
        *   **Warning:** "Override Mode Active" indicator when manual values are used.
    *   **Export:** This baseline is included in the binary header so the Runtime can apply Z-Score normalization.

*   **REQ-FORGE-006: The VAK Refinery (Visual/Auditory/Kinesthetic Synthesis).**
    *   **Goal:** Create 3 specific Reference Vectors for Visual, Auditory, and Kinesthetic modalities. The Runtime uses these to identify the user's preferred learning/communication style.
    *   **Problem:** We don't have dedicated "Visual learner" audio files. VAK profiles must be synthesized from archetype compositions.
    *   **Method:** Use **Archetype Composition** with hardcoded psychology rules based on VAK theory:

    *   **VAK Profile Definitions:**

        | VAK Type | Semantic Source | Chronemic Profile | Tone Profile | Rationale |
        |:---|:---|:---|:---|:---|
        | **Visual** | Centroid of Innovation + Power | WPM = `Mean + (1.5 * StdDev)` (Fast) | High Energy | Visualizers speak quickly, use "see/look/picture" language |
        | **Auditory** | Centroid of Passion + Prestige | WPM = `Mean` (Moderate) | Clear, Resonant | Auditory learners match pace, use "hear/sound/listen" language |
        | **Kinesthetic** | Centroid of Trust + Mystique | WPM = `Mean - (1.0 * StdDev)` (Slow) | Warm, Low | Kinesthetics speak slowly, use "feel/grasp/touch" language |

    *   **Synthesis Algorithm:**
        ```rust
        fn synthesize_vak_profile(
            vak_type: VakModality,
            archetypes: &HashMap<Advantage, NorthStarReferenceSet>,
            baseline: &CulturalBaseline,
        ) -> NorthStarReferenceSet {
            match vak_type {
                VakModality::Visual => {
                    // Semantic: Average of Innovation + Power
                    let semantic = average_vectors(
                        &archetypes[&Advantage::Innovation].semantic,
                        &archetypes[&Advantage::Power].semantic,
                    );
                    // Chronemic: Fast pace
                    let chronemic = [
                        baseline.mean_wpm + (1.5 * baseline.std_dev_wpm), // Fast WPM
                        baseline.mean_latency - (0.5 * baseline.std_dev_latency), // Short pauses
                        baseline.mean_jitter, // Normal jitter
                        0.0, // No overlap
                    ];
                    // ... compose other streams
                }
                VakModality::Auditory => {
                    // Semantic: Average of Passion + Prestige
                    let semantic = average_vectors(
                        &archetypes[&Advantage::Passion].semantic,
                        &archetypes[&Advantage::Prestige].semantic,
                    );
                    // Chronemic: Moderate pace
                    let chronemic = [
                        baseline.mean_wpm, // Average WPM
                        baseline.mean_latency, // Average pauses
                        baseline.mean_jitter * 0.8, // Slightly lower jitter (rhythmic)
                        0.0,
                    ];
                }
                VakModality::Kinesthetic => {
                    // Semantic: Average of Trust + Mystique
                    let semantic = average_vectors(
                        &archetypes[&Advantage::Trust].semantic,
                        &archetypes[&Advantage::Mystique].semantic,
                    );
                    // Chronemic: Slow pace
                    let chronemic = [
                        baseline.mean_wpm - (1.0 * baseline.std_dev_wpm), // Slow WPM
                        baseline.mean_latency + (1.0 * baseline.std_dev_latency), // Long pauses
                        baseline.mean_jitter * 0.5, // Low jitter (steady)
                        0.0,
                    ];
                }
            }
        }
        ```

    *   **UI Control: "VAK Tuner" Panel:**
        *   Three expandable sections (Visual / Auditory / Kinesthetic).
        *   **Source Archetypes:** Dropdown to change which archetypes compose the VAK profile (e.g., swap "Innovation" for "Alert" in Visual).
        *   **Speed Offset Slider:** Adjust the WPM multiplier (default: +1.5σ for Visual, 0 for Auditory, -1.0σ for Kinesthetic).
        *   **Preview Button:** Plays a synthetic audio sample at the calculated WPM/Latency to "hear" the profile.
        *   **Lock Toggle:** Once satisfied, lock the VAK profile to prevent accidental changes.

    *   **Validation:**
        *   After synthesis, verify that the 3 VAK vectors have cosine distance > 0.3 from each other (they must be distinct).
        *   If VAK profiles overlap too much, UI displays warning: *"VAK differentiation too low. Adjust speed offsets or source archetypes."*

    *   **Benefit:** Creates 3 scientifically-grounded VAK North Stars that the Runtime uses for modality identification, synthesized from your archetype training data without requiring separate VAK-specific audio.

### 3.3 Module C: The Proving Ground (Testing & Validation)
**Goal:** Verify the coordinate system works on new data.

*   **REQ-TEST-001: Mystery Injection.**
    *   Allow upload of a "Test File" (Audio or Text) that was *not* in the training set.
*   **REQ-TEST-002: Live Classification.**
    *   System embeds the test file and calculates Cosine Similarity against all 49 generated North Stars.
    *   *Visual:* A ranked list of the Top 5 matches with confidence scores.
*   **REQ-TEST-003: VAK Validation.**
    *   Dedicated test zone for Visual/Auditory/Kinesthetic text inputs to verify VAK vector separation.
*   **REQ-TEST-004: The Ghost Protocol (Auto-Validation).**
    *   **Problem Solved:** Manually testing one file at a time is slow. You need to know: "If I ran this on 100 random people, how often would it be right?"
    *   **Solution:** Leave-One-Out Cross-Validation (LOOCV).
    *   **Action:** When clicked, the system runs a simulation:
        1.  It temporarily removes **10%** of your training data from the centroid calculation.
        2.  It calculates North Stars using the remaining **90%**.
        3.  It treats the removed 10% as "Unknown Users."
        4.  It attempts to classify them.
    *   **Output:** **Confusion Matrix.**
        *   *"When the input was actually 'Power', the system guessed 'Power' 95% of the time and 'Prestige' 5% of the time."*
    *   **Value:** Ultimate proof that your system is working perfectly before you ever run the real app.
    *   **Benefit:** Simulates thousands of tests to prove reliability.
*   **REQ-TEST-004a: Tile-Based Cross-Validation & CUDA Graph Execution.**
    *   **Feature Used:** *Nsight Compute Tile Profiling* and *CUDA Graphs with Device-Side Node Profiling*.
    *   **Implementation:** The "Ghost Protocol" (LOOCV) requires training and testing 100+ times rapidly. This simulation loop is written as a custom **CUDA Tile Kernel**.
    *   **CUDA Graph Capture:** Capture the entire validation loop (Split Data → Calc Reference Vector → Classify → Record Score) into a single **CUDA Graph**. Upload the Graph to the GPU once. Trigger it with a single CPU signal.
    *   **Benefit:** Reduces the "Ghost Protocol" runtime from minutes to seconds. The validation runs at "Hardware Speed" with zero CPU intervention. The progress bar zips from 0% to 100% almost instantly, making stress testing feel like a lightweight operation.

### 3.4 Module D: Visualization & Analytics
**Goal:** See the math.

*   **REQ-VIZ-003: Cluster Health Telemetry (Silhouette Score).**
    *   **Problem Solved:** Looking at 3D dots is subjective. You need a number that proves your North Stars are distinct.
    *   **Solution:** The **Silhouette Coefficient** measures how similar an object is to its own cluster (cohesion) compared to other clusters (separation).
    *   **Metric: Global Separation Score (0.0 - 1.0).**

*   **REQ-VIZ-004: Ghost Overlay (A/B Version Comparison).**

#### 3.5 Module E: The Export & Serialization Engine
**Goal:** Serialize the calculated state into the strict binary format required by the runtime, including all **52 Reference Sets** (49 Archetypes + 3 VAK) and the **Cultural Baseline** for Z-Score normalization.

*   **REQ-EXP-001: Strict Schema Compliance (Updated for 6-Stream + VAK)**
    *   The system MUST serialize the data into a `HogsheadBinaryPackage` struct containing:
        1.  `archetypes: HashMap<u8, NorthStarReferenceSet>` - The 49 Hogshead Archetypes.
        2.  `vak_profiles: HashMap<VakModality, NorthStarReferenceSet>` - The 3 VAK Reference Sets.
        3.  `baseline: CulturalBaseline` - The cultural baseline statistics for Z-Score normalization.
        4.  `schema_version: u8` - Version 2 (added Chronemic + VAK support).
    *   The serialization format MUST be **Little Endian** (standard for x86_64/Bincode) to ensure compatibility with the Runtime architecture.
    *   **Constraint:** The `NorthStarReferenceSet` struct definition in this tool MUST be byte-for-byte identical to the struct in the Runtime crate (`cognitive-core`), including the new `chronemic: [f32; 4]` field.

*   **REQ-EXP-002: Pre-Flight Integrity Check (Updated for 52 Arrays)**
    *   Before allowing export, the system MUST run a comprehensive validation pass:
        1.  **Archetype Count Check:** Are there exactly 49 keys in the archetypes HashMap? (Panic if < 49).
        2.  **VAK Count Check:** Are there exactly 3 keys in the vak_profiles HashMap (Visual, Auditory, Kinesthetic)? (Panic if < 3).
        3.  **6-Stream Vector Check:** For ALL 52 Reference Sets:
            *   Semantic: 768 dimensions, non-zero values.
            *   Tone: 512 dimensions, non-zero values.
            *   Emotion: 1024 dimensions, non-zero values.
            *   Style: Scalar value (can be 0.0-1.0).
            *   Action: 42 dimensions, valid probability distribution (sums to ~1.0).
            *   **Chronemic: 4 dimensions, non-zero values** (NEW).
        4.  **Cultural Baseline Check:**
            *   `mean_wpm` must be between 80-220 (reasonable speech range).
            *   `std_dev_wpm` must be > 0 (avoid division by zero in Runtime).
            *   `sample_count` must be > 0.
        5.  **VAK Orthogonality Check:**
            *   Minimum angular distance between any two VAK profiles must be > 30°.
            *   If < 30°: Block export with error *"VAK profiles too similar. Runtime will misclassify users."*

*   **REQ-EXP-003: Atomic Write**
    *   The file export should write to a temporary file first (e.g., `.tmp`) and rename it to `hogshead_reference_vectors.bin` only upon successful completion to prevent file corruption.

*   **REQ-EXP-004: Binary Schema Version Header (New)**
    *   The binary file MUST begin with a 4-byte header:
        *   Bytes 0-1: Magic number `0x4847` ("HG" for Hogshead).
        *   Byte 2: Schema version (currently `0x02` for v2 with Chronemic + VAK).
        *   Byte 3: Reserved for future flags.
    *   **Purpose:** Allows the Runtime to detect and reject incompatible binary files.
    *   **Backward Compatibility:** Runtime should gracefully error if schema version < 2: *"Binary file missing Chronemic/VAK data. Please regenerate with North Star Refinery v2.0+"*

*   **REQ-EXP-005: Export Summary Report (New)**
    *   Upon successful export, generate a human-readable summary file (`hogshead_reference_vectors.json`) containing:
        ```json
        {
            "schema_version": 2,
            "export_timestamp": "2025-12-06T08:15:00Z",
            "archetype_count": 49,
            "vak_count": 3,
            "cultural_baseline": {
                "mean_wpm": 145.2,
                "std_dev_wpm": 28.5,
                "mean_latency_ms": 450.0,
                "std_dev_latency_ms": 180.0,
                "sample_count": 847
            },
            "vak_orthogonality": {
                "visual_auditory_angle": 65.2,
                "auditory_kinesthetic_angle": 58.7,
                "kinesthetic_visual_angle": 72.1,
                "minimum_angle": 58.7,
                "status": "PASS"
            },
            "checksum_sha256": "a1b2c3d4e5f6..."
        }
        ```
    *   **Purpose:** Allows verification without loading the binary file.

#### 3.6 Module F: The Math Wizard (Auto-Optimization)
**Goal:** Automatically find the mixing weights that maximize the distance between different archetypes.

*   **REQ-OPT-001: The "Maximize Separation" Button**
    *   **Problem:** You don't know if the Primary/Secondary mix should be 70/30, 65/35, or 60/40.
    *   **Action:** When clicked, the system runs a **Grid Search**.
    *   **Logic:**
        1.  It simulates mixing the archetypes at every increment (0.60, 0.61, 0.62... up to 0.90).
        2.  For each increment, it calculates the **Global Silhouette Score** (how distinct the 49 points are).
        3.  It selects the specific weight that produces the highest score.
    *   **Result:** The system automatically sets the sliders to the mathematically perfect ratio. You do zero guessing.

*   **REQ-OPT-002: Dynamic Contrast Enhancement (The "Squash" Function)**
    *   **Problem:** Raw vectors are often "fuzzy" and bunched together.
    *   **Action:** Add a "Contrast" slider that applies a mathematical **Sigmoid** function to the final vectors.
    *   **Math:** $Vector_{new} = \frac{1}{1 + e^{-k(Vector_{old})}}$ (where $k$ is contrast).
    *   **Effect:** This pushes values closer to 0 or 1, making the vectors "sharper" and easier for the runtime to classify.
    *   **UX:** You drag one slider called "Contrast," and the clusters on the screen physically move apart.

*   **REQ-OPT-003: Custom Metric Sandbox (cuTile Python).**
    *   **Feature Used:** *cuTile Python (Numba JIT Compiler)*.
    *   **The Problem:** Right now, the PRD hardcodes "Cosine Similarity" and "Weighted Mixing." As a Data Scientist, you might want to experiment: *"What if I use Mahalanobis distance for the Tone stream? What if I use a non-linear mixing function?"* You shouldn't have to recompile Rust code to test a math hypothesis.
    *   **UI:** A "Scripting Console" tab in the Math Wizard.
    *   **Action:** Allows you to write a Python function for vector mixing or distance calculation.
    *   **Backend:** The system JIT-compiles this Python code into **CUDA Tile IR** at runtime and executes it on the GPU.
    *   **Benefit:** You get Python's ease of use with the RTX 5090's raw hardware speed. You can prototype complex mixing logic instantly without touching the Rust codebase.

#### 5.1.2.1 Nomic Embed Text v1 (Stream A: Semantic)

**CRITICAL: Task Instruction Prefixes (Mandatory)**

| Use Case | Prefix | Example |
|:---|:---|:---|
| **Document Embedding** | `search_document: ` | `search_document: The user expressed confidence in the proposal.` |

```python
from sentence_transformers import SentenceTransformer
import torch.nn.functional as F

model = SentenceTransformer("nomic-ai/nomic-embed-text-v1", trust_remote_code=True)

# For archetype training data (use search_document prefix)
texts = ["search_document: I demand results immediately."]
embeddings = model.encode(texts)

# CRITICAL: L2 normalize for cosine similarity
embeddings = F.normalize(torch.tensor(embeddings), p=2, dim=1)
```

**For Long Context (8192 tokens):**
```python
from transformers import AutoTokenizer, AutoModel

tokenizer = AutoTokenizer.from_pretrained('bert-base-uncased', model_max_length=8192)

    'nomic-ai/nomic-embed-text-v1',
    trust_remote_code=True,
    rotary_scaling_factor=2  # Required for 8192 context
)
```

**Output Processing:**
1. Extract hidden states from model output
2. Apply mean pooling across token dimension
3. **L2 normalize** the final embedding (mandatory for cosine similarity)

```python
# All audio MUST be resampled to 16kHz before inference
```

```python
from transformers import Wav2Vec2FeatureExtractor, WavLMForXVector
import torch

feature_extractor = Wav2Vec2FeatureExtractor.from_pretrained('microsoft/wavlm-base-plus-sv')

# Process audio (must be 16kHz)
inputs = feature_extractor(

    sampling_rate=16000,
    padding=True,
```

#### 5.1.2.3 Wav2Vec2 Emotion (Stream C: Emotion)

**Source:** [wav2vec2-emotion](https://huggingface.co/models?search=wav2vec2+emotion) (local fine-tuned model)
**Base Architecture:** wav2vec2-large (facebook/wav2vec2-large-960h or similar)
**Paper:** "wav2vec 2.0: A Framework for Self-Supervised Learning of Speech Representations" (arXiv:2006.11477)

| Specification | Value |
|:---|:---|
| **Hidden State Dimension** | **1024** (wav2vec2-large architecture) |
| **Transformer Layers** | 24 |
| **Sampling Rate** | **16kHz (CRITICAL)** |
| **Input Type** | Raw audio waveform (float array) |
| **Output** | Pooled hidden states for emotion representation |
| **License** | MIT (base model) |

**Architecture Note:**
The PRD specifies using wav2vec2-large (1024-dim) rather than wav2vec2-base (768-dim) or pruned variants. The larger model provides richer emotional feature extraction critical for distinguishing archetype communication patterns.

| Variant | Hidden Size | Layers | Use Case |
|:---|:---|:---|:---|
| wav2vec2-base | 768 | 12 | Lightweight, faster inference |
| **wav2vec2-large** | **1024** | **24** | **North Star Refinery (recommended)** |
| wav2vec2-large-robust-12 | 768 | 12 | Pruned, emotion-specific |

**CRITICAL: The NorthStarReferenceSet struct requires 1024-dim emotion vectors. Use wav2vec2-large.**

**Implementation:**
```python
from transformers import Wav2Vec2Processor, Wav2Vec2Model
import torch

# Use wav2vec2-large for 1024-dim output
processor = Wav2Vec2Processor.from_pretrained('facebook/wav2vec2-large-960h')
model = Wav2Vec2Model.from_pretrained('/path/to/wav2vec2-emotion')  # Fine-tuned local model

# Audio must be 16kHz
inputs = processor(audio_array, sampling_rate=16000, return_tensors="pt", padding=True)

with torch.no_grad():
    outputs = model(**inputs)
    # Use last_hidden_state for embeddings
    hidden_states = outputs.last_hidden_state  # [batch, seq, 1024]
    # Mean pooling across time dimension
    embeddings = hidden_states.mean(dim=1)  # [batch, 1024]
    # L2 normalize
    embeddings = torch.nn.functional.normalize(embeddings, dim=-1)
```

**Emotion-Relevant Features Captured:**
- **Prosodic contours:** Pitch variation, rhythm, tempo
- **Spectral characteristics:** Voice quality, breathiness, tension
- **Temporal dynamics:** Pauses, speech rate changes, emphasis patterns

#### 5.6.1 REQ-CUDA-001: Green Context Partitioning (Spatial Isolation)

*   **Feature Used:** *Green Contexts (Runtime API Exposure)*.
*   **Problem:** Calculating Reference Vectors or running UMAP for 3D visualization usually spikes GPU usage to 100%, causing the UI (rendering the 3D galaxy) to stutter or freeze.
*   **Implementation:** The Refinery runs on two distinct Green Contexts:
    1.  **Context A (Visualization Lane - 17 SMs / 10%):** Dedicated strictly to the Three.js/OpenGL rendering loop. Guarantees 60FPS UI smoothness even while heavy math runs.
    2.  **Context B (The Forge Lane - 153 SMs / 90%):** Dedicated to Batch Inference and Matrix Math.
*   **Benefit:** "Silky smooth" UI responsiveness regardless of workload.

**Implementation Pattern:**

```cpp
#include <cuda_runtime.h>

void setup_green_contexts() {
    int device_id = 0;
    cudaInitDevice(device_id);

    // Step 1: Get available SMs (170 total on RTX 5090)
    cudaDevResource_t total_sm_resource;
    cudaDeviceGetDevResource(&total_sm_resource, device_id, cudaDevResourceTypeSm);

    // Step 2: Split SMs (10% viz = 17 SMs, 90% compute = 153 SMs)
    cudaDevResource_t viz_sm_resource, compute_sm_resource;
    cudaDevResourceSMSplit(
        &viz_sm_resource,       // 10% SMs for UI
        &compute_sm_resource,   // 90% SMs for compute
        &total_sm_resource,
        0.1f                    // viz fraction
    );

    // Step 3: Create resource descriptors
    cudaDevResourceDesc_t viz_desc, compute_desc;
    cudaDevResourceGenerateDesc(&viz_desc, &viz_sm_resource, 1);
    cudaDevResourceGenerateDesc(&compute_desc, &compute_sm_resource, 1);

    // Step 4: Create green contexts
    cudaExecutionContext_t viz_ctx, compute_ctx;
    cudaGreenCtxCreate(&viz_ctx, viz_desc, device_id, 0);
    cudaGreenCtxCreate(&compute_ctx, compute_desc, device_id, 0);

    // Step 5: Create streams for each context
    cudaStream_t viz_stream, compute_stream;
    cudaStreamCreateWithEC(&viz_stream, viz_ctx, cudaStreamDefault, 0);
    cudaStreamCreateWithEC(&compute_stream, compute_ctx, cudaStreamDefault, 0);

    // Now launch work on separate streams - they run in parallel without contention!
    // Visualization (OpenGL/Three.js rendering)
    render_3d_galaxy<<<grid_viz, block_viz, 0, viz_stream>>>(viz_args);

    // Heavy computation (batch inference, UMAP, etc.)
    batch_inference_kernel<<<grid_compute, block_compute, 0, compute_stream>>>(compute_args);
}
```

**Key API Functions:**
- `cudaDeviceGetDevResource()` - Get available SMs from device
- `cudaDevResourceSMSplit()` - Partition SMs by fraction
- `cudaDevResourceGenerateDesc()` - Create resource descriptor
- `cudaGreenCtxCreate()` - Provision green context
- `cudaStreamCreateWithEC()` - Create stream bound to context

#### 5.6.2 REQ-CUDA-002: Grouped GEMM for Hexa-Modal Inference

*   **Feature Used:** *cuBLAS Grouped GEMM (Blackwell Exclusive)*.
*   **Problem:** You are running 5 different models (Nomic, WavLM, Wav2Vec2, RoBERTa, BERT). Traditionally, these run sequentially (Serial Stream).
*   **Implementation:** Use Grouped GEMM to fuse the linear algebra layers of these disparate models into single kernel launches where possible.
*   **Benefit:** Increases batch ingestion speed by ~4x. You can drop 1,000 files and they process instantly.

**Implementation Pattern (49 Secondary Archetype Mixing):**

```cpp
#include <cublas_v2.h>

void mix_all_secondaries_grouped(
    cublasHandle_t handle,
    float *d_primaries,       // [7][768] - 7 primary archetypes
    float *d_secondaries,     // [49][768] - 49 secondary outputs
    float primary_weight,     // 0.7
    float secondary_weight    // 0.3
) {
    int num_groups = 49;

    // Setup arrays for grouped operation
    std::vector<cublasOperation_t> transa(num_groups, CUBLAS_OP_N);
    std::vector<cublasOperation_t> transb(num_groups, CUBLAS_OP_N);
    std::vector<int> m(num_groups, 768);  // vector dimension
    std::vector<int> n(num_groups, 1);
    std::vector<int> k(num_groups, 1);
    std::vector<float> alpha(num_groups, primary_weight);
    std::vector<float> beta(num_groups, secondary_weight);
    std::vector<int> lda(num_groups, 768);
    std::vector<int> ldb(num_groups, 768);
    std::vector<int> ldc(num_groups, 768);
    std::vector<int> group_sizes(num_groups, 1);

    // Setup pointers for each combination (7x7 = 49)
    std::vector<const float*> A_ptrs(num_groups);
    std::vector<const float*> B_ptrs(num_groups);
    std::vector<float*> C_ptrs(num_groups);

    int idx = 0;
    for (int i = 0; i < 7; i++) {
        for (int j = 0; j < 7; j++) {
            A_ptrs[idx] = d_primaries + i * 768;  // Primary A
            B_ptrs[idx] = d_primaries + j * 768;  // Primary B
            C_ptrs[idx] = d_secondaries + idx * 768;
            idx++;
        }
    }

    // SINGLE kernel launch for ALL 49 operations!
    cublasSgemmGroupedBatched(
        handle,
        transa.data(), transb.data(),
        m.data(), n.data(), k.data(),
        alpha.data(),
        A_ptrs.data(), lda.data(),
        B_ptrs.data(), ldb.data(),
        beta.data(),
        C_ptrs.data(), ldc.data(),
        num_groups,
        group_sizes.data()
    );
}
```

#### 5.6.4 REQ-CUDA-004: CUDA Tile Programming for Vector Mixing

*   **Feature Used:** *CUDA Tile IR / cuTile Python*.
*   **Problem:** Traditional SIMT programming requires manual thread management for vector operations. Tile programming abstracts this complexity.
*   **Implementation:** Use CUDA Tile IR or cuTile Python for high-level tile-based operations that automatically utilize Tensor Cores.

**Tile IR Conceptual Model:**
```
Traditional SIMT:  Thread → operates on single element
Tile Programming:  Tile Kernel → operates on entire tile (multi-dimensional array)
```

**cuTile Python Implementation (REQ-OPT-003 Custom Metric Sandbox):**

```python
import cuda.tile as ct
import cupy as cp

TILE_SIZE = 768  # Semantic vector dimension

@ct.kernel
def cosine_distance_kernel(vec_a, vec_b, result):
    """
    User-defined distance metric executed on GPU via Tile IR.
    JIT-compiled to native Blackwell instructions.
    """
    bid = ct.bid(0)

    # Load 768-dim vectors as tiles
    a = ct.load(vec_a, index=(bid,), shape=(TILE_SIZE,))
    b = ct.load(vec_b, index=(bid,), shape=(TILE_SIZE,))

    # Cosine similarity (uses Tensor Cores automatically)
    dot_product = ct.sum(a * b)
    norm_a = ct.sqrt(ct.sum(a * a))
    norm_b = ct.sqrt(ct.sum(b * b))
    similarity = dot_product / (norm_a * norm_b)
    distance = 1.0 - similarity

    # Store result
    ct.store(result, index=(bid,), tile=distance)

def compute_distances(vectors_a: cp.ndarray, vectors_b: cp.ndarray) -> cp.ndarray:
    """Host-side launch function."""
    num_pairs = vectors_a.shape[0]
    result = cp.empty(num_pairs, dtype=cp.float32)

    grid = (ct.cdiv(num_pairs, 1), 1, 1)
    ct.launch(cp.cuda.get_current_stream(), grid, cosine_distance_kernel,
              (vectors_a, vectors_b, result))

    return result
```

```cuda
tile<2x128x128xf16>    // batched 2D matrices (FP16)
```

#### 5.6.5 REQ-CUDA-005: cuFFT Device Extensions (Zero-Copy Audio)

*   **Feature Used:** *cuFFTDx (Device-Side FFT)*.
*   **Problem:** Traditional audio processing requires CPU-GPU round-trips for spectrogram extraction.
*   **Implementation:** Execute FFT directly inside CUDA kernels, enabling Load → FFT → VAD → Store in a single kernel.

**Implementation Pattern:**

```cpp
#include <cufftdx.hpp>

// Define FFT type (1024-point R2C, Blackwell SM 100)
using FFT_1K = decltype(
    cufftdx::Size<1024>() +
    cufftdx::Type<cufftdx::fft_type::r2c>() +
    cufftdx::Direction<cufftdx::fft_direction::forward>() +
    cufftdx::Precision<float>() +
    cufftdx::SM<100>()  // Blackwell compute capability
);

__global__ void audio_preprocessing_kernel(
    float *audio_files,        // [N][samples] - batch of audio (GPU resident)
    float *spectrograms,       // [N][freq_bins] - output spectrograms
    float *vad_energy,         // [N] - voice activity energy per file
    int num_files,
    int samples_per_file
) {
    int file_id = blockIdx.x;
    if (file_id >= num_files) return;

    extern __shared__ char smem[];

    // Load audio frame
    float *audio_frame = audio_files + file_id * samples_per_file;

    // FFT buffer in registers
    FFT_1K::value_type fft_buffer[FFT_1K::storage_size];

    // Copy audio to FFT buffer
    for (int i = threadIdx.x; i < 1024; i += blockDim.x) {
        fft_buffer[i] = audio_frame[i];
    }
    __syncthreads();

    // Execute FFT in device code (no CPU round-trip!)
    FFT_1K().execute(fft_buffer, smem);
    __syncthreads();

    // Compute magnitude spectrogram and VAD energy
    float energy_sum = 0.0f;
    for (int i = threadIdx.x; i < 512; i += blockDim.x) {
        float real = fft_buffer[i].x;
        float imag = fft_buffer[i].y;
        float magnitude = sqrtf(real * real + imag * imag);

        spectrograms[file_id * 512 + i] = magnitude;
        energy_sum += magnitude;
    }

    // Warp reduction for VAD energy
    energy_sum = warpReduceSum(energy_sum);
    if (threadIdx.x == 0) {
        vad_energy[file_id] = energy_sum;
    }
}
```

**Benefit:** Audio preprocessing adds <1ms per file. No CPU involvement after initial GPU upload.

#### 5.6.6 REQ-CUDA-006: Deterministic Reductions (CCCL 3.1)

*   **Feature Used:** *CCCL 3.1 Deterministic Reductions*.
*   **Problem:** Standard parallel reductions are non-deterministic due to floating-point non-associativity. Running centroid calculation twice may produce different results.
*   **Implementation:** Use `cub::DeviceReduce` with `run_to_run` determinism flag.

**Determinism Modes:**
| Mode | Guarantee | Performance | Use Case |
|:---|:---|:---|:---|
| `not_guaranteed` | None (fastest) | Baseline | Prototyping |
| `gpu_to_gpu` | Same GPU, bitwise identical | 1.2x slower | Production |
| **`run_to_run`** | **Across runs/GPUs, bitwise identical** | **1.5x slower** | **North Star Refinery** |

**Implementation Pattern:**

```cpp
#include <cub/device/device_reduce.cuh>

void calculate_centroid_deterministic(
    float *d_samples,      // [num_samples][dim] - all samples
    float *d_centroid,     // [dim] - output centroid
    int num_samples,
    int dim                // 768 for semantic, 512 for tone, etc.
) {
    for (int d = 0; d < dim; d++) {
        // Extract dimension d from all samples
        float *d_dim_values;
        cudaMalloc(&d_dim_values, num_samples * sizeof(float));
        extract_dimension<<<...>>>(d_samples, d_dim_values, d, dim, num_samples);

        // Deterministic sum with run-to-run guarantee
        float *d_sum;
        cudaMalloc(&d_sum, sizeof(float));

        void *d_temp = nullptr;
        size_t temp_bytes = 0;

        // Get temp storage size
        cub::DeviceReduce::Sum(d_temp, temp_bytes, d_dim_values, d_sum, num_samples);
        cudaMalloc(&d_temp, temp_bytes);

        // Execute with CRITICAL determinism flag
        cub::DeviceReduce::Sum<
            float*, float*, cub::Sum,
            cub::DeterminismMode::run_to_run  // Bitwise reproducible
        >(d_temp, temp_bytes, d_dim_values, d_sum, num_samples);

        // Compute mean
        float sum_host;
        cudaMemcpy(&sum_host, d_sum, sizeof(float), cudaMemcpyDeviceToHost);
        float mean = sum_host / num_samples;
        cudaMemcpy(d_centroid + d, &mean, sizeof(float), cudaMemcpyHostToDevice);

        cudaFree(d_dim_values);
        cudaFree(d_sum);
        cudaFree(d_temp);
    }
}
```

**Result:** North Star coordinates are **mathematically absolute constants** (bitwise identical across runs).

#### 5.6.7 REQ-CUDA-007: CUDA Graphs with Device-Side Launch

*   **Feature Used:** *CUDA Graphs + Device-Side Graph Launch*.
*   **Problem:** Ghost Protocol (LOOCV) requires running 100+ validation iterations. Sequential kernel launches create CPU bottleneck.
*   **Implementation:** Capture validation workflow as CUDA Graph, launch from device kernel.

**Graph Capture:**

```cpp
cudaGraph_t validation_graph;
cudaGraphExec_t validation_exec;
cudaStream_t temp_stream;
cudaStreamCreate(&temp_stream);

// Begin capture
cudaStreamBeginCapture(temp_stream, cudaStreamCaptureModeGlobal);

// Capture validation workflow (these are NOT executed yet)
split_data_kernel<<<grid, block, 0, temp_stream>>>(split_args);
calc_reference_kernel<<<grid, block, 0, temp_stream>>>(ref_args);
classify_samples_kernel<<<grid, block, 0, temp_stream>>>(classify_args);
record_accuracy_kernel<<<grid, block, 0, temp_stream>>>(accuracy_args);

// End capture
cudaStreamEndCapture(temp_stream, &validation_graph);

// Instantiate (optimize for execution)
cudaGraphInstantiate(&validation_exec, validation_graph, nullptr, nullptr, 0);
```

**Device-Side Launch (CUDA 13.1 Feature):**

```cpp
// Device kernel that launches graphs without CPU involvement
__global__ void ghost_protocol_device_runner(
    cudaGraphExec_t graph,
    int num_iterations,
    float *d_accuracy_scores
) {
    for (int i = threadIdx.x; i < num_iterations; i += blockDim.x) {
        // Launch graph FROM DEVICE - no CPU round-trip!
        cudaGraphLaunch(graph, 0);

        // Record accuracy (in-kernel)
        d_accuracy_scores[i] = /* ... */;
    }
}

// Host launch
cudaGraphExec_t *d_graph;
cudaMalloc(&d_graph, sizeof(cudaGraphExec_t));
cudaMemcpy(d_graph, &validation_exec, sizeof(cudaGraphExec_t), cudaMemcpyHostToDevice);

ghost_protocol_device_runner<<<1, 100>>>(*d_graph, 100, d_scores);
cudaDeviceSynchronize();  // Completes in SECONDS instead of minutes!
```

**Limitation:** Maximum 120 graphs submitted per device.

## 6. Implementation Guide: The Fine-Tuning Loop

This is the specific workflow for the Data Scientist using the tool.

### Step 1: Baseline Generation
1.  Load 20 generic "Power" text sentences.
2.  Load 20 generic "Power" audio clips.
3.  Click **Forge**.
4.  Go to **Visualize**. Look at the "Power" cluster.
5.  *Observation:* The audio cluster is too spread out (Standard Deviation > 0.5).
6.  *Action:* Go back to **Sources**. Identify the 3 audio clips that are outliers (poor recording quality or wrong tone). Delete them.
7.  Click **Forge** again.
8.  *Result:* Cluster tightens. Centroid is now accurate.

### Step 2: Mixing Calibration
1.  Look at the derived archetype "The Catalyst" (Passion + Innovation).
2.  In the **Visualize** tab, "The Catalyst" is overlapping heavily with "The Rockstar" (Innovation + Passion).
3.  *Diagnosis:* The distinction between Primary and Secondary is too weak with a 70/30 split.
4.  *Action:* Adjust **Mixing Slider** to 80/20.
5.  *Result:* The two archetypes separate in vector space. The heatmap turns from Red to Green.

### Step 3: VAK Tuning
1.  Load "Visual" text samples.
2.  Click **Test**. Upload a sentence: "I see what you mean."
3.  *Result:* System classifies as Auditory.
4.  *Diagnosis:* The Semantic keyword weights are off.
5.  *Action:* Manually adjust the **Semantic Weight** in the settings or add more specific "Visual" training data to pull the centroid.
